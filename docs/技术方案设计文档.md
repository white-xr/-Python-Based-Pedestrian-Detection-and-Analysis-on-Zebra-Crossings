# 技术方案设计文档

## 项目名称
《基于YOLOv8算法的斑马线行人检测与安全预警系统》

## 文档目标
本方案聚焦代码实现与算法设计，重点明确系统的技术架构、模块职责、算法流程、性能优化策略与阶段性落地路径。目标是在保证检测与预警效果的同时，尽可能将实时性能提升到 30 FPS 左右。

## 一、技术目标
- 实现斑马线场景下的行人实时检测
- 实现斑马线区域检测与空间关系分析
- 实现行人轨迹跟踪与状态建模
- 引入 LSTM 进行行人未来轨迹预测
- 基于预测结果实现多级安全预警
- 构建可视化清晰、便于调试和答辩展示的界面
- 通过多线程、异步流水线、低拷贝传输等方式优化性能
- 在目标硬件条件下尽量达到 30 FPS
- 最低保证实时可用帧率不低于 20 FPS

## 二、总体技术架构
建议整体采用“分层 + 异步流水线”的结构。

1. 数据输入层
- 视频文件读取
- 摄像头采集
- 帧缓存管理

2. 感知层
- 行人检测
- 斑马线检测
- 可选车辆检测

3. 时序分析层
- 多目标跟踪
- 轨迹平滑
- 速度/方向估计
- LSTM 轨迹预测

4. 风险决策层
- 行人与斑马线关系判断
- 过街意图识别
- 风险评分与分级预警

5. 展示与输出层
- 检测框绘制
- 轨迹/预测轨迹绘制
- 风险颜色渲染
- 日志、状态栏、FPS 显示
- 录制输出

## 三、核心算法方案

### 3.1 行人检测
采用 YOLOv8 作为主检测器。

建议方案：
- 默认模型：`yolov8n` 或 `yolov8s`
- 若设备有较好 GPU，可切换更高精度模型
- 仅保留 `person` 类别
- 配置项可调：`conf_threshold`、`iou_threshold`、`imgsz`、`device`

建议优化：
- 检测输入尺寸优先控制在 `416` 或 `480`
- 对非关键帧允许复用上帧检测结果
- 推理前尽量避免额外颜色空间重复转换

### 3.2 斑马线检测
斑马线检测建议采用双方案设计。

方案 A：YOLO 斑马线检测
- 使用专门训练的斑马线模型
- 复杂场景表现更强

方案 B：传统视觉检测兜底
- 灰度化
- 边缘检测
- 形态学闭运算
- 轮廓筛选
- 长宽比和面积过滤

最终建议：
- 默认优先使用 YOLO 斑马线检测
- 模型不可用或效果不稳定时自动回退到传统方法
- 对斑马线检测结果做缓存与平滑，避免闪烁

### 3.3 多目标跟踪
第一阶段建议使用轻量 IOU 跟踪，优先保证实时性。

每个目标维护：
- `track_id`
- `bbox`
- `history`
- `velocity`
- `speed`
- `missed_frames`

第二阶段可增强：
- 引入 `ByteTrack` 或 `SORT`

### 3.4 关键点定义
为了让轨迹更符合过街行为，建议使用脚点而不是框中心点。

定义：
- `foot_x = (x1 + x2) / 2`
- `foot_y = y2`

原因：
- 与斑马线的实际接触位置更接近
- 用于判断是否进入斑马线更合理
- 更适合作为 LSTM 输入

### 3.5 轨迹预测
LSTM 的核心作用是提前判断行人的未来运动趋势。

输入：
- 同一行人最近 `T` 帧脚点序列
- 建议 `T = 8~12`

输出：
- 未来 `K` 帧位置序列
- 建议 `K = 5~15`

推荐结构：
- 输入维度：2
- hidden dim：32 或 64
- 层数：1 到 2
- 输出层：全连接层映射到 `(x, y)`

预测方式：
- 若 LSTM 权重存在，走 LSTM 预测
- 若权重不存在，使用常速度模型兜底

### 3.6 过街意图识别
建议采用“特征 + 规则”的方式，保证可解释性。

输入特征：
- 当前脚点到斑马线区域的距离
- 行人速度大小
- 行人运动方向是否指向斑马线
- 最近若干帧轨迹是否持续靠近斑马线
- 预测轨迹是否会进入斑马线区域

输出状态：
- 无过街意图
- 疑似过街意图
- 明确过街意图

### 3.7 风险评估与预警
预警层建议采用“评分 + 分级”的混合机制。

风险因子：
- 当前是否位于斑马线
- 距离斑马线边界的远近
- 运动方向是否朝向斑马线
- 当前速度
- 预测轨迹进入斑马线的时间
- 可选：车辆接近状态

风险等级：
- 安全
- 注意
- 危险

建议规则：
- 安全：当前不在斑马线，且预测轨迹不进入斑马线
- 注意：靠近斑马线，或预测轨迹将在短时间进入斑马线
- 危险：已位于斑马线，或预测轨迹明确进入斑马线核心区域

建议拆分：
- `Tracker`

### 4.4 风险分析模块
职责：
- 斑马线关系判断
- 意图识别
- 风险分级

建议拆分：
- `IntentAnalyzer`

- 框绘制

## 五、实时性与性能优化方案

### 5.1 性能目标
- 理想目标：30 FPS
- 可接受目标：25 FPS
- 最低下限：20 FPS

### 5.2 异步流水线设计
建议至少拆成 3 个线程：

1. 采集线程
- 读取摄像头或视频帧
- 放入输入队列

2. 推理线程
- 执行检测、跟踪、预测、预警
- 放入结果队列

3. UI 渲染线程
- 只显示最新结果
- 避免 UI 积压

### 5.3 队列策略
建议使用“有界队列 + 丢旧保新”：
- 输入队列满时优先丢弃旧帧
- 显示队列始终只保留最新处理结果

### 5.4 低拷贝与零拷贝思路
严格零拷贝在 Python + OpenCV + PyQt 中不一定完全做到，但应尽量接近。

优化方向：
- 尽量复用 `numpy` 数组
- 减少 `frame.copy()`
- 只有在显示或叠加绘制时才生成展示帧
- `QImage` 构造时尽量直接引用连续内存
- 避免 BGR/RGB 多次来回转换

### 5.5 推理优化
建议：
- 模型初始化只做一次
- 使用 `model.fuse()`
- 控制 `imgsz`
- 降低检测频率，跟踪补帧
- 斑马线检测使用低频刷新加缓存
- 预测模块只对有效轨迹执行

推荐策略：
- 行人检测每 `2~4` 帧一次
- 斑马线检测每 `10~30` 帧一次
- 中间帧使用跟踪和缓存结果补偿

### 5.6 绘制优化
建议：
- 控制显示分辨率
- 减少复杂透明混合次数
- 复杂视觉效果做成可开关
- Debug 模式显示全部信息
- Demo 模式只显示核心信息

### 5.7 FPS 监控
系统需实时显示：
- 当前 FPS
- 平均 FPS
- 单帧处理耗时
- 检测耗时
- 绘制耗时

## 六、界面可视化技术方案

### 6.1 展示目标
界面要做到：
- 一眼看懂系统在做什么
- 风险状态足够直观
- 适合截图放论文
- 能显示性能指标

### 6.2 核心显示元素
必须显示：
- 原始画面
- 处理后画面
- 行人检测框
- 斑马线区域
- 行人 ID
- 行人轨迹
- 预测轨迹
- 当前人数
- 风险等级
- 风险原因
- FPS

### 6.3 视觉规范
建议：
- 绿色：安全
- 橙色：注意
- 红色：危险
- 斑马线：半透明荧光区域
- 预测轨迹：虚线
- 当前轨迹：实线
- 高危目标使用更粗边框或闪烁标记

### 6.4 日志展示
建议输出结构化语句，例如：
- `Track 3: 靠近斑马线，预测2.0s内进入，风险=注意`
- `Track 5: 已进入斑马线，风险=危险`

## 七、训练与数据方案

### 7.1 轨迹数据来源
可选方案：
- 从检测与跟踪结果中自动抽取轨迹
- 手工筛选有效行人轨迹
- 保存为序列坐标数据

### 7.2 数据格式
每条样本可定义为：
- 输入：过去 `T` 帧脚点坐标
- 标签：未来 `K` 帧脚点坐标

### 7.3 训练目标
损失函数可使用：
- `MSE Loss`

评价指标可使用：
- 平均位置误差
- 最终位置误差

如果时间紧，也可以先不做完整训练，用：
- 常速度预测作为 baseline
- LSTM 作为增强实验

## 八、阶段性实现路线

### 阶段一：稳定主链路
- 重构检测、跟踪、界面主流程
- 修复编码与日志问题
- 保证基础检测稳定运行

### 阶段二：预警基础版
- 完成行人与斑马线关系判断
- 加入风险分级
- 完成基本可视化展示

### 阶段三：LSTM 预测版
- 接入轨迹序列输入
- 实现 LSTM 或占位预测器
- 加入提前预警逻辑

### 阶段四：性能冲刺版
- 多线程重构
- 队列化异步处理
- 低拷贝优化
- FPS 优化，目标逼近 30 FPS

## 九、主要技术风险
- 斑马线检测在复杂光照下可能不稳定
- LSTM 训练数据不足时可能预测不准
- UI 绘制可能成为隐藏性能瓶颈
- Python 多线程在 CPU 密集型场景下受限，需要区分 I/O 与推理任务
- 纯 CPU 环境下稳定 30 FPS 难度较高，需要模型和绘制双优化

## 十、推荐的最终技术落地版本
最推荐的代码算法方案是：
- YOLOv8 行人检测
- YOLO/传统方法双路径斑马线检测
- 轻量多目标跟踪
- 脚点轨迹建模
- LSTM + 常速度兜底轨迹预测
- 基于规则和评分的三级预警
- 多线程异步流水线
- 面向 30 FPS 的性能优化
